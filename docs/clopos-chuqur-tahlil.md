# Глубокое исследование системы Clopos (ELITE POS) — La Limonariya

**Дата:** 2026-06-13 · **Заведение:** ЛИМОНАРИЯ, Навои (venue_id = 1) · **Источник:** полный живой API-экспорт, 20 JSON-файлов в `/Users/wer/projects/la_limonariya/docs/clopos-api/`

Этот документ — синтез по всем подсистемам действующей кассовой системы ресторана. Цель — точно понять, что Clopos умеет и чего в нём нет, чтобы спроектировать наше новое приложение (склад + обвалка + себестоимость + POS).

---

## 1. Обзор

Clopos (бренд «ELITE POS») — облачная ресторанная POS-система, на которой La Limonariya работает сегодня. Это полноценная касса: каталог блюд и товаров, печать заказов по кухонным станциям, финансовый журнал с балансовыми счетами, долги клиентов, столы и залы с планом зала, роли и PIN-логин. **Но в нём полностью отсутствует учёт себестоимости и склада** — ровно та дыра, ради которой строится наше приложение.

**Три ключевых вывода:**

1. **Себестоимости нет.** Только **3 из 232** продаваемых позиций и **5 из 57** ингредиентов имеют `cost_price > 0`. У 229/232 позиций `gross_margin` захардкожен на 100%, `markup_rate = 0`. `inventory_behavior = 0` у **всех** — касса не списывает склад вообще. «Прибыль» в отчётах (маржа 97.2% за 30 дней) — фикция.
2. **Click и долг (карз) уже работают** в живой системе — это не будущая разработка. Из наших четырёх типов оплаты (нал | Click | Payme | карз) отсутствует только **Payme**. Книгу долгов надо **копировать** из модели Clopos, а не изобретать.
3. **Реальность сложнее фото-допущений:** 6 станций печати, сервис-сбор зависит от зала (10/10/0/**15%**), граница дня — **06:00**, 8 терминалов, и T3 в API активен (а не «выключен»).

## 2. Модель данных

Ядро Clopos — **единая таблица «продукт»** с дискриминатором `type`:

| type | Кол-во | Что это | Нужна техкарта? |
|---|---:|---|---|
| `DISH` | 64 | Готовится на кухне, привязана к станции | Да (в нашем приложении) |
| `GOODS` | 168 | Покупной перепродаваемый товар | Нет |
| `INGREDIENT` | 57 | Сырьё (мясо-туши, овощи, специи) | — |

Все три делят ~50 общих полей: `id`, `name/full_name/slug`, `price`, `cost_price`, `gross_margin`, `markup_rate`, `unit_id`, `category_id` (+ вложенный объект), `station_id` (+ вложенный), `parent_id` (везде `null` → плоское меню, нет вариантов/модификаторов), `sold_by_weight` (везде `false`), `inventory_behavior` (везде `0`), `hidden/status`, `deleted_at` (soft delete), e-menu поля, CDN-картинки, `taxes[]`.

### Справочники-оси (взаимно независимы)

- **CATEGORY (10)** — nested-set (`_lft/_rgt`), но фактически все depth-0. Назначение — группировка в меню.
- **STATION (6)** — маршрут печати на кухонный принтер. `printable = 1` у всех.
- **UNIT (6)** — pcs / l / m / kg / g / ml.

> **Важно:** `category_id` и `station_id` — две **независимые** FK. Маршрут печати задаётся per-item, а не выводится из категории.

### Деньги — журнал двойной записи

- **BALANCE ACCOUNT (6):** Cash, Card, Bank, Safe, Supplier (id 1000), Customer (id 1001).
- **FINANCE_TRANSACTION:** на каждом движении `before_amount` / `after_amount` (бегущий остаток), `group` (md5, связывает ноги операции), `category_id`, `payment_method_id`, `balance_id`, `receipt_id`, `sale_type_id`, `hall_id`, `table_id`, `customer_id`, `user_id` (кассир), `staff_id` (официант), `shift_date`.
- **PAYMENT_METHOD (5):** Cash(1), Card(2), Customer balance/карз(3), Cashback(4 — отключён), Click(1000).
- **SALE_TYPE (3):** зал(1), Delivery(2 — не используется), Собой(3).

### Клиент / зал / стол / терминал / пользователь

CUSTOMER → один CUSTOMER-баланс (долг = знаковый остаток). HALL (4) с per-hall сервис-сбором. TABLE (46) с координатами плана. TERMINAL (8). USER (15), PIN-логин, одна роль на пользователя.

## 9. Подсистема: Долги (карз)

**Файлы:** `customers.json` (1), баланс id 1001.

Единственный клиент **Рустам** (id 1, создан 2026-01-25): `spent = 75 900`, `receipt_count = 6`, `balance_id = 1001`. CUSTOMER-баланс = **−45 900** → должен заведению. Разрыв 75 900 − 45 900 = 30 000 (уже оплачено) — данными не объясняется.

**Модель долга = отрицательный CUSTOMER-баланс**, а не список открытых счетов. Один знаковый running-баланс на клиента. Частичные погашения поддерживаются автоматически (любая сумма двигает баланс к нулю). Лимита нет. Идентификация — по имени (телефон/email пустые). Поля loyalty/bonus/cashback присутствуют, но не используются — **drop** в нашем приложении.

> В экспорте 300 транзакций нет ни одной долговой ноги (`customer_id = null` везде) — механика проводки **выведена**, не наблюдалась. Нужен реальный пример или тест.

## 10. Что это значит для нашего приложения

### Переиспользовать из модели Clopos

- Единый продукт с дискриминатором `type` (DISH/GOODS); `type` решает, нужна ли техкарта.
- Две независимые FK: `category_id` (меню) и `station_id` (печать).
- Финансы как **журнал балансовых счетов** с `before/after_amount` (инвариант `after == before + amount`).
- Долг как **отрицательный CUSTOMER-баланс** + иммутабельный леджер.
- Сервис-сбор **per-hall** (10/10/0/15), граница дня **06:00**.
- 4 роли, PIN-логин, атрибуция продажи waiter_id + cashier_id, **без ЗП/чаевых**.
- План зала: столы с hall_id, shape, x/y/w/h. Станции печати с IP/MAC, CP866, width 48.

### Сделать иначе — это наша ценность

- **Склад и обвалка**, которых у Clopos нет: туши → части (шашлык, шапок, думба, суяк, фарш, чарви) как складские SKU с весовым балансом и %потерь.
- **Реальная себестоимость**: цена туши размазана на выход с учётом потерь (пример PLAN: 100 кг @ 50 000 с 10% потерь → 55 556/кг). Не наследовать пустой/мусорный cost Clopos.
- **Техкарты**, привязанные к DISH → продажа списывает ингредиенты/части. Приоритет — ядро расхода (ФАРШ, КУСКОВОЙ, Марварид = 4 787 шт/30д).
- **Ad-hoc производство** Шапок→Фарш как свободный документ.
- Добавить **Payme** (единственный недостающий способ оплаты).

### Интеграция с Clopos: НЕТ

API доступен (мы выгрузили 20 файлов), но его модель пустая там, где нам важно. Использовать экспорт как **сид каталога**, а не рантайм-зависимость. Цены продажи — из живого API (фото-проверены). `clopus-menu.json` оставить только ради карты IP-принтеров/терминалов.
