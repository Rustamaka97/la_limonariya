## 3. Талаб — тармоқ / локация қулфи

**Ходим тизимни фақат ресторан тармоғида ишлатсин** — ўз интернетида, уйдан эмас.
**Бозорчи — истисно** (бозордан ишлайди). Иккала эски+янги талаб битта ечим билан:
«интернет узилса ишласин» ✅ + «ходим ўз интернетида ишламасин» ✅.

### 3 қатлам (клиентда ЭМАС — СЕРВЕРда / тармоқда)

1. **LAN-only** — серверни интернетга умуман очмаслик (фақат LAN + Tailscale). Телефон мобил
   интернетда серверга **йўл тополмайди** = топология, ёрилмас (текширув эмас).
2. **Роль×манба гейт (сервер)** — staff (5 роль) фақат LAN `192.168.1.0/24`; `buyer` LAN+Tailscale.
   **Ҳақиқий peer IP**'дан. CIDR — env'да. (Wi-Fi SSID текшириш ишончсиз — «серверга уланадими» сигнали.)
3. **Offline-қулф экран** — LAN серверга тирик уланмаса → қулф. Айниқса мутация (заказ/тўлов) уланиш
   талаб қилсин. Қисқа wifi-блип кечирилади. GPS = де-приоритет (сохталашади; фақат бозорчи
   харид-исботида).

**Бозорчи:** телефонда Tailscale (VPN, аутентификацияланган — спуфланмас) + минимал ACL/ҳуқуқ.

### ⚠️ КРИТИК топилма — XFF спуфинги

Ҳозирги `clientIp()` (`apps/api/src/rate-limit.ts`) **X-Forwarded-For'нинг чап** қийматини олади.
Буни мижоз **сохталаштиради**: масофадаги ходим `X-Forwarded-For: 192.168.1.50` юборса — гейт
«LAN'да» деб ўйлайди, **қулф ёрилади**. Rate-limit'га майли, access-гейтга ҳалокатли.
**Ечим:** гейт ҳақиқий peer IP'дан (Caddy XFF'ни `remote_host` билан **қайта ёзсин**, қўшмасин),
ёки умуман public'га **очма** (Tailscale ишлат).

## 4. Threat model — таҳдид × назорат

Легенда: 🟢 бор · 🟡 қисман · 🔴 йўқ.

| # | Таҳдид | Назорат | Ҳолат |
|---|---|---|---|
| 1 | Ходим уйдан/масофадан кириши | LAN-only + роль×IP гейт | 🟡 режада |
| 2 | Меҳмон wifi'дан серверга/принтерга | Тармоқ сегментацияси (POS VLAN) | 🔴 **йўқ ⚑** |
| 3 | Wifi'да PIN/сессия сниффи | HTTPS-on-LAN (локал CA) | 🔴 HTTP |
| 4 | Хато роль endpoint чақириши | Серверда роль-гейт (`protectedProcedure`) | 🟢 **бор** |
| 5 | PIN брутфорс | scrypt хэш + LAN-гейт + уриниш-қулф | 🟡 lockout йўқ |
| 6 | DB ўғирланса PIN қайтариш | `pinLookup` pepper билан | 🟡 текшир |
| 7 | Йўқолган телефон | қисқа сессия + авто-логаут + де-провижн | 🟡 қисман |
| 8 | Ходим фирибгарлиги (void/скидка) | audit_log + катта void'га тасдиқ | 🟡 audit бор |
| 9 | База/бэкап ўғирланиши | диск шифр + шифрланган бэкап + restore тест | 🔴 йўқ |
| 10 | Сервер ўлиши (SPOF) | UPS + бэкап + захира-ПК режаси | 🔴 йўқ |

**Приоритет (сениор триаж):**
- **Лаунчгача ШАРТ:** тармоқ сегментацияси (#2) · LAN-гейт (#1) · PIN lockout (#5)
- **Тез кейин:** HTTPS-on-LAN (#3) · бэкап+диск-шифр (#9) · UPS (#10)
- **Давомий:** audit immutability (#8) · pepper (#6) · сессия/де-провижн (#7)

**Синтез:** заиф 4-рақ PIN'ни айнан **LAN-гейт + сегментация** ёпади → улар шартсиз мажбурий.
Пойдевор мустаҳкам (роль-гейт ✅, scrypt ✅, audit ✅); устига сегментация + HTTPS + lockout
қўшилса — банк даражаси.

## 5. Эртага режа — ишга тушириш

1. **Сервер тайёр** (блокер) — POS (API+веб+Postgres) серверда (`.15` ёки Шерхон айтган қути),
   миграциялар (0056/0057 гача) қўлланган.
2. **Тармоқ уланиши** — телефон `Limanariya` wifi'да серверни кўрсин (`http://192.168.1.15`).
3. **Ходимларни яратиш** — ҳар роль учун ходим + **PIN** (PIN'ни Рустам ака қўяди).
4. **Ҳар роль синови** — реал телефонда логин → таблар → асосий амал (официант заказ→принтер,
   кассир чек, бозорчи харид, директор ҳисобот).
5. **Илова ўрнатиш синови** — «главный экранга қўш» → иконка → тўлиқ экран.
6. **Тезкор синов (ихтиёрий)** — сервер бўлмаса staging (Tailscale/Cloudflare туннель) орқали.

## 7. Деплой андозаси (staging'да исботланган) + тузатишлар

> Манба: `docker-compose.yml` · `docker/deploy.sh` · `Caddyfile`.
> Staging тирик: `100.124.38.91:8082/api/health` → `200 {"ok":true}`.

### Стек (docker-compose — 5 сервис)

| Сервис | Нима |
|---|---|
| **postgres** | база (17-alpine), **ички** — host'га чиқмайди |
| **api** | tRPC (`:3000` ички), `PIN_PEPPER`, `DATABASE_URL` |
| **web** | Caddy — SPA + `/trpc`,`/api` proxy (`WEB_PORT`) |
| **sites** | лендинг + QR-меню (POS'дан алоҳида) |
| **backup** | ҳар кеча 03:00 `pg_dump.gz`, 14 кун сақлаш |

`./docker/deploy.sh`: бэкап → build → up → healthcheck → **авто-rollback** (200 бермаса эски образга).

### Threat-model тузатиши (андоза кўрилди — эски "гаплар" аслида ҳал)

- ✅ **#6 PIN pepper — БOР** (`PIN_PEPPER` env). Прод'да кучли қиймат қўй (default «dev-pepper-change-me»).
- ✅ **#9 Backup — БOР** (тунги `pg_dump` + деплой-олди encrypted `BACKUP_PASSPHRASE`). Off-site нусха қолди.
- ✅ **Postgres ички** (host port-map йўқ) → «5432 конфликти / 5433'га map» хавотир КЕРАК ЭМАС.
- ⚠️ `api:3000` staging'да фош — прод `.15`да **фақат web порт** очилсин; api + postgres ички.
- 🔴 **XFF фикс аниқ:** `Caddyfile` `reverse_proxy api:3000` XFF'ни қайта ёзмайди (Caddy default = қўшади)
  → чап-XFF мижоз назоратида. **Фикс:** `header_up X-Forwarded-For {remote_host}` (роль×IP гейт спуфланмасин).

### `.15`га деплой (қадамлар)

1. `.15`га **Docker** ўрнат.
2. Репо + **`.env`**: кучли `POSTGRES_PASSWORD`, `PIN_PEPPER`, `BOOTSTRAP_DIRECTOR_PIN` (Рустам PIN),
   `WEB_PORT`, `BACKUP_PASSPHRASE`.
3. `./docker/deploy.sh` → build + up + healthcheck.
4. web портни LAN'га боғла; **XFF фикс** + роль×IP гейт; api/postgres ички.
5. Телефон → `http://192.168.1.15:<WEB_PORT>` → PWA.

Мавжуд `.15` Postgres'га тегмайди (Docker ички namespace). Ягона очиқ савол: `.15` Postgres'да CloPOS
базаси борми (`psql -U postgres -l` — Рустам/Шерхон, credential) — CloPOS **булутда**, шунга эҳтимол йўқ.
