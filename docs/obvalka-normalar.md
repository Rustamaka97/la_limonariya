# Обвалка-нормы La Limonariya — отчёт по историческим записям

## 1. Методология

- Источник: 30 фото рукописных записей, OCR → 80 блоков.
- **Только 50 блоков** — это приходная обвалка (есть вес туши `carcass_weight>0` и разбивка по частям). Из них **qўй 28, mol 20**.
- Остальные ~30 блоков (фото `13.27.37` и далее) — **производственные ведомости остатков** (`фарш / мол-ш / буй-ш / корейка / мар-д`, вес туши = 0). Это учёт полуфабрикатов по дням, **не обвалка**, поэтому в нормы не входят (но дают ценный сигнал для модуля производства).
- Имена частей нормализованы к каноническим (см. `part_name_map`): `шопок-к / шапок к → Шапок`, `вагора/вағора → Ваггора`, `казон-к/бозон-к → Базон-к`, и т.д.
- Для расчёта **% от туши** брались только блоки, где баланс сходится (`|потеря| ≤ 30%` и сумма частей > 0): **чистых qўй 24, mol 12**. Аномальные блоки исключены (раздел 4).
- Центр нормы надёжнее брать по **медиане**, а не среднему (рукописные данные, выбросы).

## 4. Анализ потерь (yo`qotish)

- **Главный вывод: при корректной записи реальной потери при обвалке практически НЕТ.** Медиана баланса: qўй **0.9%**, mol **0.4%** — то есть сумма частей ≈ вес туши.
- Баланс сходится (±5%): qўй **16 из 24**, mol **7 из 12**.
- Технологический отход (мусор/без/буриган): qўй ≈ **1.7%** туши (медиана 1.4%); по mol почти не пишут.
- Видимые большие «потери» в отдельных блоках — **не настоящий yo`qotish**, а: (а) часть мяса сразу ушла «в фарш / салатга / кухню» отдельной строкой без веса; (б) OCR не сложил слагаемые `a+b`.
- **Рекомендация для приёмки:** норматив отхода qўй мусор 1–3%; общий допуск «несведённого» баланса ±5%; больше — флаг на проверку.

## 5. Аномалии (parts-sum ≠ carcass)

- **31.05.26 mol 95.9 кг** — сумма 155.9 кг (+62.6%): «филе 78» нереально (вероятно 7.8/17.8). OCR.
- **10-май mol Маруф 44 кг** — сумма 70.2 кг (+59.4%): «тушонка 28» нереально.
- **6 май mol Маруф 112 кг** — сумма 58.8 кг (−47.5%): известный кейс Maruf — здесь **недозапись**, а не +13.9% переизбыток; egasi должен подтвердить, ушла ли часть в фарш.
- **Блок без даты «шаш 16.45»** — +117.7%: 16.45 — это вес части, а не туши (неверный `carcass_weight`).
- **29-май qўй 72.6 кг** — в одной версии слагаемые `a+b` не сложены (сумма = 34.6); в другой версии всё сходится (−1.1%). Дубликат одной туши.
- **24.05 mol 71.7 кг** — три расходящиеся версии одной туши (low conf).

## 6. Что перепроверить с владельцем

Список в `low_confidence_blocks` — фото/даты с зачёркиваниями, несведёнными слагаемыми и нереальными числами. Приоритет: 31.05.26 mol («филе 78»), 10-май mol Маруф, 6 май Маруф 112 кг, и все три версии 24.05 mol 71.7.

## 7. Как приложение использует нормы (auto-flag)

1. При вводе новой обвалки: для каждой части считаем `% = кг_части / вес_туши`.
2. Сравниваем с **рабочей полосой** по типу туши (qўй/mol) из таблиц 2–3.
   - Внутри полосы → норма.
   - Вне полосы → **жёлтый флаг** «отклонение от нормы», показываем ожидаемый диапазон.
3. Контроль баланса: `вес_туши − Σ частей`. Если `|потеря| > 5%` → флаг «баланс не сходится» (либо забыли строку, либо ошибка веса).
4. Отход (мусор) выше 3% по qўй → флаг.
5. Нормы **самообучаемые**: каждая подтверждённая владельцем обвалка добавляется в выборку, медиана и полосы пересчитываются (растёт n → сужается полоса). Аномалии в обучение не берём.

## 8. Честно об ограничениях OCR

Данные рукописные, много `confidence: medium/low`, зачёркивания и несведённые слагаемые. Поэтому: нормы для частей с `n ≤ 2` (Буғлама, Бўйин, Вырезка, Балаза, Мастава-qўй, Умакай-qўй) — **предварительные**, нужны новые замеры. Полосы % намеренно шире медианы, чтобы не давать ложных флагов на старте. По мере накопления подтверждённых данных точность вырастет.
